# Vue3 + ExpressJS + SQLite Fullstack Setup for AI Studio & Cloud Run

This document explains the architecture and setup required to run a fullstack Vue 3 + ExpressJS + SQLite3 application that works perfectly in both the Google AI Studio preview environment and a deployed Google Cloud Run production environment.

## 1. Project Architecture

The project is a monolithic repository containing both the frontend (Vue 3 + Vite) and the backend (Express + better-sqlite3) in the same directory. The Node.js `server.ts` handles the API routes and serves the frontend.

- **Frontend:** Vue 3, Vue Router, TailwindCSS, Vite
- **Backend:** Express 5.x, TypeScript
- **Database:** better-sqlite3 (SQLite)
- **Execution:** `tsx` allows seamless development and production execution of TypeScript.

## 2. Package.json Scripts

The `package.json` relies on specific scripts to handle both development and production efficiently:

```json
"scripts": {
  "dev": "tsx server.ts",
  "build": "vite build",
  "start": "tsx server.ts",
  "preview": "vite preview",
  "lint": "eslint . --ext .js,.ts,.vue"
}
```

**Why `tsx` in production?**
Attempting to bundle an Express server with `esbuild` or `webpack` can cause "Dynamic require" errors (such as `Error: Dynamic require of "node:events" is not supported`). By using `tsx server.ts` as the production `start` script, we execute the raw TypeScript code. This avoids all bundling side-effects for the backend, running seamlessly in ES module mode while letting Vite handle the frontend dist creation.

## 3. Server Configuration (`server.ts`)

The Express server handles serving the Single Page Application (SPA) natively using different logic depending on the environment context (`NODE_ENV`):

### Development Mode (`vite middleware`)
When running `npm run dev`, Vite creates a server middleware that dynamically builds and serves the Vue app. The Express app seamlessly mounts this middleware alongside your `/api` routes, providing Hot Module Replacement (HMR).

### Production Mode (`dist/` from Cloud Run)
When published to Cloud Run, `NODE_ENV` is set to `production`. The backend relies on the `dist` folder generated by the build part (`npm run build`).

> **CRITICAL SETUP FOR EXPRESS 5.x:** When defining the catch-all route for your frontend Router (such as Vue Router), do **NOT** use the string `'*'` (`app.get('*')`). Express 5 uses a newer `path-to-regexp` module which throws a `Missing parameter name at index 1: *` error. Instead, use the RegExp format `/.*/`.

```typescript
import path from 'path';
import express from 'express';
import { fileURLToPath } from 'url';

const __dirname = path.dirname(fileURLToPath(import.meta.url));
const isProd = process.env.NODE_ENV === 'production' || process.env.PROD === 'true';

// Using tsx, __dirname represents project root directory safely
const rootDir = __dirname;
const app = express();

// ... define API routes ...

if (!isProd) {
  // Development: Use Vite middleware
  const { createServer: createViteServer } = await import('vite');
  const vite = await createViteServer({
    server: { middlewareMode: true },
    appType: 'spa',
    root: process.cwd(),
  });
  app.use(vite.middlewares);
} else {
  // Production: Serve static assets built by Vite
  const distPath = path.resolve(rootDir, 'dist');
  app.use(express.static(distPath));
  
  // Express 5.x compatible catch-all for Vue Router
  app.get(/.*/, (req, res) => {
    res.sendFile(path.join(distPath, 'index.html'));
  });
}
```

## 4. SQLite3 Database Setup

The database functionality is powered by `better-sqlite3`. Since better-sqlite3 requires native C++ bindings, completely avoiding a backend bundler (like `esbuild`) and using `tsx` stops architecture mismatch or binding failures.

```typescript
import Database from 'better-sqlite3';
import path from 'path';

// Always resolve DB relative to the project root
const dbPath = path.join(process.cwd(), 'cms.db');
const db = new Database(dbPath);

// Create required tables
db.prepare('CREATE TABLE IF NOT EXISTS users (id INTEGER PRIMARY KEY, ...)').run();
```

## 5. ESLint + Globals Packages

When configuring ESLint Flat Config (`eslint.config.js`) for full-stack environments, the linter will warn about missing variables like `document`, `window`, or `console` inside Vue and TS files. Ensure the `globals` package is installed and implement it to inject node and browser global definitions:

```javascript
// eslint.config.js
import globals from "globals";
import js from "@eslint/js";
import vue from "eslint-plugin-vue";
import ts from "typescript-eslint";

export default [
  js.configs.recommended,
  ...ts.configs.recommended,
  ...vue.configs["flat/recommended"],
  {
    languageOptions: {
      globals: {
        ...globals.browser,
        ...globals.node,
      },
    },
  },
  {
    files: ["**/*.vue"],
    languageOptions: {
      parserOptions: {
        parser: ts.parser,
      },
    },
  },
  {
    ignores: ["dist/**", "node_modules/**"],
  },
];
```

## Checklist before Publishing
1. `"start"` script in `package.json` is cleanly set to `tsx server.ts`.
2. `"build"` script is set to `vite build` (No esbuild on server).
3. `app.get(/.*/)` replaces `app.get('*')` for handling SPA routes in Express >= 5.x.
4. SQLite references the local applet root via `process.cwd()`.
5. Database migrations/statements don't rely on specific backend paths (using ES modules without `__dirname` context everywhere).
